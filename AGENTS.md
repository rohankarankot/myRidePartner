# Repository Context

## Overview
- This repo contains multiple apps sharing one backend and one user database.
- The backend is the source of truth for authentication, user identity, and app-source registration.
- Current user-facing app sources are `myridepartner` and `interport`.

## Shared Auth Model
- Users are shared across apps through the `User` table.
- Allowed apps are stored in `AppSource`.
- Per-user app usage is stored in `UserAppSource`.
- Login responses return a JWT containing the active `source`.
- Authenticated backend requests expose the active app as `req.user.authSource`.

## Multi-App Login Rules
- Every client app should send its app slug as `source` during login.
- Google login payload:
```json
{
  "token": "google_id_token",
  "source": "myridepartner"
}
```
- Password login payload:
```json
{
  "email": "user@example.com",
  "password": "secret",
  "source": "interport"
}
```
- Source values should be lowercase slugs only.

## Prisma Schema Workflow
- Do not treat `backend/prisma/schema.prisma` as the primary editable file.
- The source of truth lives in `backend/prisma/schema/`, split by feature:
- `00-base.prisma`
- `10-auth.prisma`
- `20-user-profile.prisma`
- `30-trips.prisma`
- `40-chat-notifications.prisma`
- `50-moderation.prisma`
- `backend/prisma/schema.prisma` is generated by `npm run prisma:merge`.

## Prisma Commands
- Merge schema parts: `npm run prisma:merge`
- Push schema to DB: `npm run db:push`
- Build backend: `npm run build`
- Seed current app sources: `npm run seed:app-sources`
- Register a future app source: `npm run register:app-source -- <slug> "<Display Name>"`

## Backoffice Admin Bootstrap
- Run these from `backend/` when provisioning local backoffice access.
- Create or refresh the default super admin: `npm run create:super-admin`
- List current super admins: `npm run list:super-admins`

## App Source Management
- Existing seeded app sources:
- `myridepartner`
- `interport`
- New apps should be registered through the provided npm script, not by hardcoding enum values.

## Google Auth Notes
- Backend should support multiple Google client IDs through `GOOGLE_CLIENT_IDS`.
- Use a comma-separated env value when multiple apps have different Google OAuth client IDs.

## Expectations For Future Agents
- Preserve the shared-user model unless explicitly asked to separate users per app.
- Prefer app registration through `AppSource` data, not schema enums.
- When updating Prisma models, edit files in `backend/prisma/schema/` and then run the merge workflow.
- Avoid reintroducing backoffice-specific source logic into the shared multi-app auth flow unless requested.
